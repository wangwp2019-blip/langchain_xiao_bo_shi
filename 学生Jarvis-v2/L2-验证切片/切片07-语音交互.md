# 切片07 · 语音交互（build 切片）

> **类型**：build 切片。落地 L0 能力契约 **C-IO 多模态 I/O** 中的「语音输入 + 语音输出」，挂到切片06 的聊天入口上。
> **本文件不得重定义 L0 能力契约。** 仅在聊天入口加「说/听」两条通道，**复用既有 TTS 资产**，不改对话/记忆/学情逻辑。

---

## 0. 要解决的问题

> 二年级孩子打字慢、不爱打字；纯文字入口（切片06）门槛仍高。语音是更自然的载体：**孩子说话提问/答题、Jarvis 用声音回复**，更像真陪伴。

## 1. 假设（可证伪）

> **H7：** 给聊天页加「🎤 说」（浏览器 Web Speech 识别）和「🔊 听」（服务端 edge-TTS 朗读 Jarvis 回复）两条通道，就能让孩子用语音自然交互，**且不破坏既有能力、零重型安装**。
>
> 若识别在二年级中文/口语场景**准确率太低**（频繁听错数字/算式）、或朗读**体验差**（卡顿、读出 markdown 符号、与文字不同步），则 H7 部分证伪 → 升级方案（见 §6）。

## 2. 最小机制（build 范围）

> hermes venv 现状：仅 `edge_tts` 可用，`torch/funasr/faster_whisper/ffmpeg` 均缺。故 **ASR 走浏览器、TTS 走服务端**，零重装：

- **听（TTS，服务端）**：`student_chat.py` 新增 `POST /api/tts {text, lang}` → `voice/tts.py` 的 `edge_tts`（音色 `zh-CN-XiaoxiaoNeural`）→ 返回 `audio/mpeg`。
  - 朗读前做 **`_speakable()` 清洗**：去 markdown 强调/标题/列表符号、去 emoji、段落转停顿——避免读出"星号井号"。
- **说（ASR，浏览器）**：前端用 **Web Speech API**（`webkitSpeechRecognition`，`lang=zh-CN`）：点 🎤 → 实时识别填入输入框 → 识别完成自动发送。零服务端依赖。
- **朗读开关 🔊**：header 可切静音；说话时自动停止当前朗读（避免抢话）。

## 3. 通过标准（可观测、二元判定）

| # | 标准 | 测量方式 | 通过线 |
|---|------|----------|--------|
| 1 | **听得到** | `POST /api/tts` | 返回非空 `audio/mpeg`，浏览器能播放 |
| 2 | **朗读自然** | 听一条含 `**`/emoji 的回复 | 不读出符号、停顿合理 |
| 3 | **说得出** | 点 🎤 说一句中文 | 识别文字进输入框并自动发送 |
| 4 | **闭环顺** | 说一道题 → 听 Jarvis 回复 | 说→识别→回复→朗读 全程顺，能连续多轮 |
| 5 | **不破坏既有** | 语音轮也问"我哪没掌握" | 学情/记忆/语气与切片06 一致 |

## 4. 停止条件（R6）

- **成功即停**：标准 1-5 过 → 语音交互沉淀进 L1；与 📷（切片05）共用同一入口。
- **部分证伪即升级**：识别准确率/体验不达标 → 走 §6 升级（服务端 FunASR）。

## 5. 真实用户与周期

- 用户：孩子 `g2-stu-01`（盈熙）+ 家长。
- 周期：起服务后 1 次浏览器会话（覆盖标准 1-5），重点听**数字/算式**识别准不准。
- 记录：①能否播放 ②朗读是否自然 ③识别准确率（尤其数字）④闭环是否顺 ⑤学情是否仍注入。

## 6. 升级路径（若浏览器 ASR 不够好）

| 方案 | 说明 | 代价 |
|------|------|------|
| **当前：浏览器 Web Speech** | Chrome/Edge 自带，免装免 key | 依赖联网、Chrome；中文口语/数字偶错 |
| 升级 A：服务端 FunASR | `voice/asr_router.py`（paraformer-zh）本地识别，中文更准、可离线 | 装 torch+funasr（~2GB）+ ffmpeg + 模型下载；加 `/api/asr` 上传端点 |
| 升级 B：云 ASR API | 接 DashScope/其他流式 ASR | 需 key、计费 |

## 8. 结果（2026-06-22 回填 · 功能通过 + 体验优化）

- 结论：**✅ 通过**——真人浏览器验证「说/听」功能均正常（标准1-5 成立）。
- 证据：用户实测「功能都正常」；`/api/tts` 返回 200·audio/mpeg·41KB；🎤 识别→自动发送、回复自动朗读、🔊 开关均生效。

### 体验优化（来自用户反馈，2026-06-22）

> 反馈：①大模型返回耗时；②回答**一次性整段显示**（非流式）后才朗读，整体偏慢；③**文字已显示时，自动语音显得多余**。

| 编号 | 问题 | 处理 | 状态 |
|------|------|------|------|
| **U1** | 文字已给出仍自动朗读，啰嗦且拖慢 | **语音进→语音出，打字进→只出文字**：仅当本轮输入来自 🎤 时才自动朗读；打字时文字即时显示、不朗读 | ✅ 已改 |
| **U2** | 想听某条回复时没有入口 | 每条小贾气泡加 **🔈 按需朗读按钮**（再点暂停）；🔊 主开关改为「语音提问后是否自动朗读」 | ✅ 已改 |
| **U3** | TTS 在文字之后串行调用 | 文字到达即显示、不被 TTS 阻塞；朗读异步、发新消息/说话时自动打断 | ✅ 已改 |
| **U4** | 回答非流式（整段弹出）| 受限：`hermes chat -q` 一次性返回整段，CLI 路径拿不到增量 token。**真流式需改走 agent 流式接口（SSE）**，列为后续较大改动，本切片不做 | ⏳ 待评估 |

- 沉淀：语音交互 + 智能朗读策略（voice-in/voice-out）记入 L1（与切片06 同一入口）。
- 下一步：（可选）流式输出 U4；或接 📷（OCR/切片05）。
